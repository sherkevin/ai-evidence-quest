# 第 2 单元：工具调用

## 为什么需要工具

模型本身不会自动访问资料库、计算分数或检查引用。工具调用的意义，是把模型的语言能力和外部能力连接起来。对高中课堂来说，可以把工具理解成“智能体的手”。

## 工具设计的三个要求

| 要求 | 说明 | 课堂例子 |
| --- | --- | --- |
| 名称清楚 | 工具名要让人知道它做什么 | `检索资料`、`计算相似度`、`生成引用` |
| 输入明确 | 工具需要哪些参数 | 问题文本、资料范围、返回条数 |
| 输出可查 | 工具结果要能复盘 | 命中的资料、分数、时间、失败原因 |

如果工具设计不清楚，智能体会出现两类问题：调用了错误工具，或用错误参数调用了正确工具。

## 课堂实验

设计三个工具卡片：

1. 资料检索工具：输入问题，输出相关资料；
2. 引用检查工具：输入回答和资料，判断回答是否被资料支持；
3. 拒答判断工具：输入检索分数和风险等级，输出是否拒答。

每张卡片写清楚：

- 工具名称；
- 输入；
- 输出；
- 可能失败的情况。

## 项目实践

在浏览器实验室中，关键词匹配、相似度排序、阈值判断都可以看作工具。项目展示时，建议展示一次完整工具调用日志：

```text
问题 -> 检索工具 -> 候选资料 -> 引用检查 -> 最终回答
```

## 评价标准

一个好的工具调用设计，不是“能跑就行”，而是能解释：

- 为什么调用这个工具；
- 工具返回了什么；
- 工具结果是否足够支持回答；
- 工具失败时系统怎样处理。

## 深入理解：工具不是装饰，而是责任边界

工具调用的本质，是把模型无法稳定完成的事情交给确定性的程序或外部服务。例如查找资料、计算相似度、读取表格、检查日期、生成引用编号，这些任务如果完全依赖模型“猜”，结果会很不稳定。工具让系统有机会把“想当然的回答”变成“可复查的过程”。

在校园资料问答项目中，工具至少要满足三个要求。第一，输入清楚：工具需要什么参数，例如问题文本、资料列表、关键词数量、相似度阈值。第二，输出清楚：工具返回的是资料卡片、分数、错误信息，还是需要追问的提示。第三，失败清楚：当资料文件不存在、结果为空、分数过低或格式错误时，工具必须返回可处理的失败状态，而不是让系统继续编造。

## 工具设计表

| 工具名称 | 输入 | 输出 | 失败处理 | 课堂可观察点 |
| --- | --- | --- | --- | --- |
| 关键词提取 | 用户问题 | 关键词列表 | 关键词过少则要求补充问题 | 同一句问题能否得到稳定关键词 |
| 资料检索 | 问题、资料卡片 | 候选资料与分数 | 分数低于阈值则拒答或追问 | 排名前三条是否真的相关 |
| 引用生成 | 候选资料编号 | 带编号的引用 | 资料缺失则不生成引用 | 回答能否追溯到资料 |
| 风险判断 | 问题和回答草稿 | 风险等级 | 高风险转人工确认 | 哪些问题不能直接回答 |

这张表可以直接进入作品集。它能说明学习者并不是随便调用接口，而是在设计一个可解释的系统。

## 动手实验：让工具返回结构化结果

课堂实验可以从一个检索工具开始。不要只返回一段文字，而要返回结构化结果：

```json
{
  "query": "图书馆周末几点关门",
  "hits": [
    {
      "title": "图书馆开放时间",
      "score": 0.82,
      "source": "library-rule-2026",
      "quote": "周六、周日开放至 18:00"
    }
  ],
  "decision": "answer"
}
```

结构化结果的好处是后续步骤可以继续检查：分数是否足够高，引用是否为空，资料是否过期，是否需要追问。这样一来，工具调用不只是“能拿到结果”，而是能支持评测和调试。

## 项目加厚任务

本单元建议补充三份材料。第一份是工具清单，列出每个工具的输入、输出和失败状态。第二份是工具日志，记录至少五次真实查询的工具返回结果。第三份是工具误差分析，说明哪些查询因为关键词不准、资料太少或阈值设置不当而失败。三份材料放在作品集中，会比单纯展示页面更能体现课程的完整度。
