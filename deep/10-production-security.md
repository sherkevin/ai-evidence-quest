# 第 10 单元：生产化、安全与结课作品

## 从课堂演示到稳定项目

课堂演示能跑一次，不代表项目可靠。生产化思维要求系统具备版本、日志、评测和安全边界。

## 稳定运行清单

| 项目 | 检查问题 |
| --- | --- |
| 资料版本 | 资料什么时候更新，谁更新 |
| 参数记录 | 检索权重和拒答阈值是否记录 |
| 运行日志 | 每次问题、候选资料和回答是否可查 |
| 失败案例 | 错误是否进入改进表 |
| 人工确认 | 高风险问题是否交给人工 |

## 安全风险

可信问答项目至少要考虑：

- 提示诱导：用户要求系统忽略资料；
- 资料污染：资料库中混入错误或恶意内容；
- 越权回答：回答涉及处分、健康或个人隐私；
- 过度自动化：把辅助判断当作最终决定。

## 课堂任务

为项目写一份安全说明，包含：

1. 系统能回答什么；
2. 系统不能回答什么；
3. 哪些问题必须人工确认；
4. 如何记录和修正错误。

## 结课档案

最终提交不只是网页链接，而是一份完整档案：

- 课程问题；
- 系统结构；
- 资料样例；
- 交互页面；
- 测试问题；
- 失败案例；
- 安全边界；
- 下一步计划。

这份档案正是作品集最有价值的部分：它证明课程不是临时堆功能，而是从知识教学到项目实践的完整设计。

## 深入理解：上线思维不是企业专属

即使课程项目不会真正投入学校使用，也应该训练上线思维。上线思维不是追求复杂部署，而是让作品能被他人打开、理解、测试和复盘。一个只在作者电脑上能运行的项目，很难说明教学成果；一个有页面、有说明、有测试、有边界、有失败案例的项目，才像完整作品。

对于校园资料问答项目，生产化可以简化为五件事：页面可访问，资料可追溯，回答可引用，失败可复盘，风险可控制。每一件事都能对应到具体文件和展示材料。

## 安全检查清单

| 检查项 | 通过标准 |
| --- | --- |
| 资料来源 | 每张资料卡片都有来源和日期 |
| 引用格式 | 回答中的关键结论能追溯到资料 |
| 拒答规则 | 资料不足、问题模糊、高风险问题能被识别 |
| 日志记录 | 至少保留问题、命中资料、回答、失败原因 |
| 隐私保护 | 不上传个人隐私、成绩、联系方式等敏感信息 |
| 页面说明 | 明确这是学习项目，不替代正式通知或人工判断 |

这份清单可以作为结课答辩前的自查表。每一项都可以由同伴互评。

## 作品集结构

推荐最终作品集包含以下文件：

```text
project/
  README.md              # 项目说明与展示入口
  data/cards.json         # 资料卡片
  labs/playground.html    # 交互页面
  tests/questions.md      # 测试问题
  tests/failures.md       # 失败案例
  docs/system-map.md      # 系统结构图
  docs/safety.md          # 边界与安全说明
```

文件结构不必完全一致，但必须让读者能顺着入口理解项目。优秀作品不是把所有材料堆在一起，而是让别人能沿着“问题、资料、系统、实验、评测、反思”的顺序读下去。

## 结课展示脚本

展示时建议控制在五分钟左右：第一分钟说明问题背景，第二分钟展示系统结构和资料卡片，第三分钟演示成功样例，第四分钟演示失败或拒答案例，第五分钟说明下一步改进。主动展示失败案例会让项目更可信，因为它说明作者知道系统的边界。

## 项目加厚任务

最后一次提交前，检查作品集中是否存在“三个证据”：代码证据、实验数据证据、反思证据。代码证据说明作品能运行；实验数据证据说明作品被测试过；反思证据说明作者知道系统的不足并能提出改进。三类证据齐全，才是一套真正完整的人工智能课程作品。
