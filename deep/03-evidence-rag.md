# 第 3 单元：证据问答与检索增强生成

## 一句话理解

检索增强生成可以理解为“先查资料，再组织回答”。它把生成式人工智能的表达能力限制在可检查的资料范围内。

## 核心流程

```text
问题理解 -> 资料检索 -> 候选排序 -> 证据选择 -> 回答生成 -> 引用检查
```

这个流程的重点不是让回答更像人，而是让回答可以回到资料。对高中项目来说，最重要的能力是让学习者看到：答案为什么来自这些资料，而不是凭空出现。

## 资料卡片设计

每张资料卡片至少包含：

| 字段 | 作用 |
| --- | --- |
| 编号 | 用于引用 |
| 标题 | 概括资料主题 |
| 正文 | 支持回答的具体内容 |
| 标签 | 帮助检索 |
| 来源 | 说明资料从哪里来 |
| 更新时间 | 判断资料是否过期 |
| 风险等级 | 决定是否需要人工确认 |

## 实验任务

准备 12 张校园资料卡片，覆盖图书馆、社团、实验室、校历四类场景。然后设计 10 个问题：

- 4 个可以直接回答；
- 2 个问题表达模糊；
- 2 个资料不足；
- 2 个涉及安全或处分，需要人工确认。

对每个问题记录：命中资料、排序结果、回答内容、是否拒答、改进建议。

## 进阶讨论

检索增强生成仍然会出错。常见错误包括：

- 资料库没有答案；
- 检索到了相近但不支持结论的资料；
- 回答把资料外的信息写进去了；
- 引用看起来存在，但引用内容并不能证明结论。

因此本单元必须和“引用检查、拒答和评测”一起学习。

## 深入理解：检索增强生成的四个环节

检索增强生成并不是“先搜索再回答”这么简单。一个完整流程至少包括资料准备、问题表示、候选召回和证据约束回答四个环节。资料准备决定系统能不能找到答案；问题表示决定系统怎样理解提问；候选召回决定哪些资料进入模型视野；证据约束回答决定最终文本是否只使用资料中的信息。

| 环节 | 关键问题 | 课堂实验 |
| --- | --- | --- |
| 资料准备 | 资料切多大、是否带来源、是否有更新时间 | 把一份校规切成 5 张资料卡片 |
| 问题表示 | 只用关键词，还是使用语义向量 | 比较“借书数量”和“最多能借几本书” |
| 候选召回 | 取前几条资料，阈值设多少 | 调整阈值观察回答变化 |
| 证据约束 | 回答是否完全来自资料 | 给回答逐句标注依据 |

这一流程可以让学习者看到，生成模型只是最后一环。真正决定可信度的，往往是前面的资料整理和检索设计。

## 资料卡片格式

课程中建议使用统一格式保存资料卡片：

```json
{
  "id": "library-001",
  "title": "图书馆开放时间",
  "text": "图书馆周一至周五开放至 21:00，周六、周日开放至 18:00。",
  "source": "图书馆管理规定",
  "date": "2026-03-01",
  "tags": ["图书馆", "开放时间"]
}
```

这个格式刻意保留来源、日期和标签。来源用于引用，日期用于判断是否过期，标签用于检索和人工检查。即使未来接入向量数据库，也不应该丢掉这些基础字段。

## 课堂实验：构造反例

只做成功样例会让项目显得单薄。本单元必须构造反例。例如资料库中只有“图书馆周末开放到 18:00”，但问题问“寒假图书馆开放吗”。系统如果直接回答周末时间，就是把不相关资料套到了新问题上。正确做法是说明“当前资料只包含平时周末开放时间，不能推出寒假开放安排”。

建议每组至少准备三类反例：资料缺失、资料相似但不支持结论、资料过期或冲突。展示时把反例讲清楚，作品的可信度会明显提高。

## 项目加厚任务

本单元完成后，作品集中应出现一份“证据链样例”。每个样例包含原始问题、命中资料、回答草稿、引用检查、最终回答和是否拒答。证据链不是附属材料，而是可信问答项目的核心成果。它证明系统不是把问题扔给模型，而是在围绕证据工作。
