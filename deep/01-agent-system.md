# 第 1 单元：智能体系统

## 本单元问题

普通聊天机器人主要根据输入生成文字；智能体系统则需要在目标、工具、资料和环境之间循环工作。高中课堂可以先不讨论复杂框架，而是抓住一句话：

> 智能体不是一个会说话的模型，而是一个会根据目标选择行动的系统。

## 系统的四个部件

| 部件 | 课堂解释 | 校园资料问答项目中的例子 |
| --- | --- | --- |
| 目标 | 这次任务要完成什么 | 回答图书馆、社团、实验室等校园规则问题 |
| 工具 | 系统能调用什么能力 | 检索资料、计算分数、生成引用、拒答 |
| 资料 | 系统依据什么判断 | 校园规则卡片、活动通知、安全说明 |
| 记忆 | 系统要保留什么上下文 | 上一次检索到的资料、失败案例、用户补充信息 |

智能体设计的关键，是把“模型回答”变成“系统过程”。系统过程应该能被观察、记录和复盘。

## 课堂任务

选择一个校园场景，画出智能体系统图。

示例场景：

- 查询图书馆开放时间；
- 查询社团教室申请流程；
- 查询实验室安全规则；
- 查询校历中的考试安排。

要求图中必须出现：输入问题、资料库、检索器、回答生成、引用检查、拒答判断。

## 代码阅读

教师资源包中保留了智能体课程的入门代码和笔记本。阅读时不要急着运行所有代码，先观察三个问题：

1. 哪些函数是“工具”；
2. 哪些数据是“资料”；
3. 哪些步骤在记录系统状态。

## 项目产出

本单元结束时，项目档案中应增加一张“系统结构图”和一段说明：这个问答助手不是直接把问题交给模型，而是先检索资料，再根据证据回答。

## 深入理解：从“回答者”到“行动系统”

课堂上最容易出现的误解，是把智能体理解成“更会聊天的模型”。为了避免这个误解，本单元把智能体拆成一个可观察的行动系统。系统收到问题后，不能立刻给出最终回答，而要经历目标识别、资料检索、证据选择、回答生成、引用核对和风险判断。每一步都有输入和输出，也都有可能失败。

可以把这个过程写成一张状态表：

| 状态 | 系统正在做什么 | 需要留下什么记录 |
| --- | --- | --- |
| 接收问题 | 判断问题属于哪个校园场景 | 原始问题、关键词、场景标签 |
| 选择行动 | 决定调用检索、追问或拒答 | 行动原因、可选行动 |
| 获取资料 | 在资料卡片中寻找依据 | 命中的资料、相似度、来源 |
| 组织回答 | 把资料整理成自然语言 | 使用了哪些资料、哪些没有使用 |
| 检查边界 | 判断是否越界、是否需要人工确认 | 拒答原因、风险等级、改进建议 |

这张表的价值不只是帮助写代码，更是帮助学习者形成工程思维：一个系统如果不能说明自己做过什么，就很难被信任，也很难被改进。

## 代码实验：用最小代码模拟智能体循环

可以先不用任何复杂框架，只用一段伪代码理解智能体循环：

```python
def answer(question, cards):
    goal = classify_question(question)
    action = choose_action(goal)
    if action == "refuse":
        return "资料不足，暂时不能回答。"
    evidence = retrieve_cards(question, cards)
    if not enough(evidence):
        return "没有找到足够证据，建议查询教务处或相关老师。"
    draft = write_answer(question, evidence)
    return attach_citations(draft, evidence)
```

这段代码虽然简单，却包含了智能体系统最重要的骨架：分类、行动选择、资料检索、证据判断、生成回答和引用。后续单元会逐步把每个函数变得更可靠。

## 课堂组织建议

本单元适合采用“先拆系统，再画图，再写伪代码”的方式。先让学习者说出日常问答中最常见的风险，例如“问到校规没有依据”“把过期通知当成新通知”“把一个社团的规则套到另一个社团”。然后把这些风险放回系统结构图，讨论哪一步可以减少风险。最后再写最小伪代码，让系统图不只是图，而是能被实现的流程。

## 常见误区

第一种误区是只画模型，不画资料。没有资料库，可信问答就失去了依据。第二种误区是只画成功路径，不画失败路径。真实系统中，资料不足、问题模糊、引用冲突都很常见。第三种误区是把人工确认放到最后一句口号里，而不是设计成系统动作。只要涉及安全、纪律、成绩、健康、隐私等高风险问题，系统就应该明确转交人工确认。
