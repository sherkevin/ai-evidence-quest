# 第 8 单元：上下文工程与记忆

## 上下文不是越多越好

智能体系统需要上下文，但上下文过多会带来噪声、成本和风险。上下文工程要解决的问题是：哪些信息应该放进当前任务，哪些信息应该压缩、删除或转入长期记录。

## 三类上下文

| 类型 | 例子 | 处理方式 |
| --- | --- | --- |
| 当前问题 | “周日图书馆开放吗？” | 必须保留 |
| 检索资料 | 图书馆开放时间卡片 | 保留与问题直接相关的部分 |
| 会话历史 | 上一轮用户补充“我问的是高中部图书馆” | 压缩成摘要 |

## 记忆的边界

课堂项目不建议保存个人隐私信息。可以保存的是：

- 失败案例；
- 资料更新记录；
- 常见问题类型；
- 系统参数修改记录。

不应保存的是：

- 未经同意的个人身份信息；
- 心理、健康、处分等敏感判断；
- 与项目目标无关的长期聊天记录。

## 实验任务

设计一个会话摘要：

```text
用户连续问了三次图书馆问题，第二轮补充“周末”，第三轮补充“借书数量”。系统应该保留哪些信息？
```

写出摘要，并说明哪些原始信息可以丢弃。

## 项目产出

最终项目应包含“上下文策略”：一次回答最多使用几条资料，是否保留上一轮问题，失败案例怎样进入项目日志。

## 深入理解：上下文不是越多越好

很多人以为把所有资料都塞给模型，回答就会更准确。实际情况常常相反：上下文太多会让系统分心，旧信息会干扰新问题，甚至把错误资料带入下一轮回答。上下文工程的核心，是选择哪些信息应该留下，哪些信息应该丢弃，哪些信息必须被标记为不可靠。

在校园资料问答项目中，上下文至少包括四类：当前问题、候选资料、上一轮对话、系统日志。当前问题一定要保留；候选资料要按分数和来源筛选；上一轮对话只在连续追问时保留；系统日志主要用于调试，不应直接进入回答。

## 会话摘要模板

```json
{
  "topic": "图书馆借阅规则",
  "current_need": "确认周末借书数量限制",
  "known_facts": [
    "用户前一轮问过开放时间",
    "资料库包含借阅数量规定"
  ],
  "missing_facts": [
    "是否处于寒暑假特殊安排"
  ],
  "risk": "低"
}
```

摘要的作用不是复述所有对话，而是帮助系统保留与下一轮回答有关的信息。它要短、准、可检查。

## 记忆的三种用途

第一，记住任务进度。例如系统已经检索过图书馆规则，下一轮可以继续围绕同一主题查找。第二，记住失败案例。例如某类问题经常把“教室申请”和“实验室借用”混淆，后续检索可以优先检查场景标签。第三，记住用户补充的限定条件。例如“周末”“高一社团活动”“校本课展示”这些信息会改变回答范围。

但记忆也有边界。涉及个人身份、联系方式、健康、处分、成绩等敏感信息，不应该随意保留；不可靠的模型猜测不能写入长期记忆；过期资料必须标记时间。

## 项目加厚任务

本单元建议加入“上下文失败案例”。选择一个连续追问场景，分别测试“不保留上下文”“保留全部上下文”“只保留摘要”三种方式。记录哪一种更稳定，为什么。这个实验非常适合展示课程深度，因为它把抽象的上下文管理变成了可以观察的系统行为。
