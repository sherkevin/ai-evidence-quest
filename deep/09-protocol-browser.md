# 第 9 单元：协议、接口与浏览器任务

## 为什么要讲接口

当智能体只在一个网页里运行时，工具调用很简单；当它要连接搜索、文件、数据库或浏览器任务时，就需要清楚的接口。接口的意义是让系统知道：可以调用什么、需要什么输入、会得到什么输出。

## 接口说明模板

| 字段 | 说明 |
| --- | --- |
| 工具名称 | 做什么 |
| 输入参数 | 需要哪些信息 |
| 输出格式 | 返回什么结构 |
| 权限范围 | 能访问什么，不能访问什么 |
| 失败处理 | 调用失败时怎样提示 |

## 浏览器任务的课堂化处理

浏览器智能体容易显得很炫，但高中课程应把重点放在“可控任务”上。例如：

- 打开一页公开校历；
- 提取活动日期；
- 与资料卡片比对；
- 生成引用记录。

不要让系统自由浏览不确定网页，也不要处理需要登录或涉及隐私的页面。

## 课堂任务

把“查询公开活动通知”拆成三个接口：

1. 读取网页文本；
2. 提取日期和地点；
3. 写入资料卡片。

每个接口写清楚输入、输出和失败处理。

## 项目产出

本单元可作为社团拓展。结课作品可以只展示接口设计，不一定接入真实浏览器自动化；重点是让学习者知道外部工具必须有边界和记录。

## 深入理解：协议让工具协作变得可控

当系统需要读取网页、调用表格、查询日历或连接资料库时，最危险的做法是让模型自由操作一切。更可靠的做法是通过明确协议规定：系统能调用哪些工具，每个工具接受什么输入，返回什么输出，哪些动作需要人工确认，哪些日志必须保存。

协议可以理解成工具之间的“课堂规则”。如果规则不清楚，系统可能重复调用工具、读取错误页面、把临时结果当成权威资料，甚至执行不该执行的操作。规则清楚之后，模型的自由度被限制在安全范围内，系统才更容易被评测和复现。

## 接口说明模板

| 字段 | 示例 | 说明 |
| --- | --- | --- |
| 接口名称 | 读取活动通知 | 用一句话说明工具用途 |
| 输入 | 页面地址、关键词 | 必须明确类型和限制 |
| 输出 | 标题、日期、地点、正文摘要 | 不能只返回散乱文本 |
| 失败状态 | 页面打不开、信息缺失、格式错误 | 每种失败都要可处理 |
| 安全边界 | 不读取登录后页面，不保存个人信息 | 明确不能做什么 |

这张表适合放入作品集，也适合用于小组分工。负责页面的同学知道要展示什么，负责算法的同学知道要接收什么，负责评测的同学知道怎样设计失败样例。

## 浏览器任务的课堂化处理

真实浏览器自动化对环境要求较高，不一定适合作为所有班级的必做内容。高中课程中可以采用“模拟接口”的方式：教师提供几段网页文本，学习者编写提取日期、地点、对象和要求的函数。这样既能理解浏览器任务的结构，又不会被环境配置拖住。

进阶小组可以继续尝试真实网页读取，但必须遵守边界：只读取公开页面，不绕过登录，不抓取个人信息，不进行自动提交，不把未核验的信息写入正式资料库。

## 项目加厚任务

本单元建议加入一份“接口设计说明书”。它不需要很长，但必须完整：接口列表、输入输出样例、失败状态、安全边界、日志字段。展示时可以用一个活动通知作为例子，说明系统如何从网页文本中提取资料卡片，再进入检索增强生成流程。
