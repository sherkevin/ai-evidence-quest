# 第 11 章：从课堂 demo 到学校项目

## 项目边界

课堂 demo 可以使用虚构资料；学校项目必须考虑隐私、更新、权限和人工确认。不能把一个演示页面直接说成可上线系统。

## 最小可行版本

如果要把项目做成校内科创作品，建议只做四件事：

1. 上传公开资料；
2. 检索资料片段；
3. 带引用回答；
4. 资料不足或高风险时拒答。

暂时不要做：

- 自动处分判断；
- 医疗、心理、安全事故建议；
- 未成年人个人信息画像；
- 无审核的公开发布。

## 角色分工

| 角色 | 任务 |
| --- | --- |
| 资料组 | 收集和切分资料 |
| 检索组 | 设计关键词或向量检索 |
| 评测组 | 编写测试问题和 badcase |
| 展示组 | 做海报、说明文档和演示脚本 |

## 展示脚本

一个好的展示不是“看它能答什么”，而是：

1. 展示资料库；
2. 展示一次正确引用；
3. 展示一次拒答；
4. 展示一个 badcase 和改进过程；
5. 说明边界和风险。
